# opencv-object-counting

一个基于 OpenCV 的传统视觉计数实验项目，目标是对规则目标完成分割、轮廓提取、计数和结果标注。

当前分支 `archive/counting-v1` 不是稳定版本，而是一次“提升鲁棒性”的失败尝试。它从单一路径的 HSV 方案，推进到了“先判断图像类型，再决定是否做分水岭拆分”的流程化实现。代码已经能编译，也具备一定实验价值，但还没有形成可复用、可稳定迁移到新图片的方案。

## 这个分支在做什么

这次尝试想解决的问题是：

- 不同图片不一定适合同一种分割方式
- 有些图片颜色信息明显，有些主要依赖明暗
- 目标之间有时是分离的，有时存在粘连

所以当前分支实现了一条分支式流程：

1. 读取原图
2. 判断基础处理图类型
3. 用 Otsu 做初始二值化
4. 统一前景/背景极性
5. 根据轮廓面积分布判断是否可能存在粘连
6. 如果疑似粘连，则尝试用分水岭拆分
7. 提取轮廓并按面积过滤
8. 在原图上绘制外接框

对应代码主要在：

- `ObjectCounting/main.cpp`
- `ObjectCounting/Process.h`
- `ObjectCounting/Process.cpp`

## 当前流程说明

### 1. 基础图选择

`judgeBasicType()` 会先把图像转到 HSV 空间，取 S 通道均值：

- 如果平均饱和度较高，认为颜色区分更明显，走 S 通道
- 如果平均饱和度较低，退回灰度图

这一步的目标是避免所有图片都硬套同一种输入表征。

### 2. 初始二值化

`basicImg2Binary()` 对基础图执行 Otsu 阈值分割，并通过 `normalizeBinaryMask()` 尽量统一为“白色是目标、黑色是背景”的二值图，方便后续逻辑复用。

### 3. 粘连判断

`judgeCrave()` 会提取外轮廓，计算平均面积和最大面积：

- 如果最大轮廓面积明显大于平均值
- 则认为可能存在粘连目标
- 进一步尝试分水岭拆分

### 4. 分水岭拆分

`craveType()` 的流程是：

- 开运算去掉一些毛刺
- 膨胀得到更保守的背景
- 距离变换提取前景核心区域
- 连通域生成 markers
- 调用 `watershed()` 做拆分
- 最后把结果重新整理回二值 mask

### 5. 轮廓过滤与可视化

`contoursProcess()` 会：

- 从 mask 提取轮廓
- 按图像面积比例过滤小噪声
- 在原图上绘制绿色外接框

## 为什么说这次鲁棒性尝试失败了

这条路线比最初版本更“像架构”，但还没有真正解决泛化问题，核心原因大致有这几个：

- `S` 通道均值这个判断太粗。它能区分部分样本，但很难稳定区分“颜色差异主导”还是“光照差异主导”。
- Otsu 阈值对样本分布很敏感。一旦背景、反光、阴影变化变大，初始 mask 就会不稳定。
- “最大轮廓面积 / 平均轮廓面积”这个粘连判断启发式过于脆弱，目标尺寸变化、数量变化、噪声变化都会影响判断。
- 分水岭非常依赖前面的二值结果。前面一旦错，分水岭往往会把错误进一步放大，而不是自动修复。
- 当前缺少一套成体系的样例、指标和批量验证流程，所以只能凭单张效果看起来“像是可以”，很难证明方法真的稳。
- 绝对路径和本地 OpenCV 配置写死在工程里，可移植性也比较弱。

换句话说，这个分支完成的是“把问题结构化”，但没有完成“把方案做稳”。

## 构建与运行

当前工程是 Visual Studio C++ 项目，已在本机环境下验证可以成功构建：

- 工具链：MSBuild / MSVC v145
- 语言标准：C++20
- OpenCV：工程当前链接的是本地 `D:\OpenCV\opencv\build`

当前工程文件里写死了本地依赖路径：

- Include: `D:\OpenCV\opencv\build\include`
- Lib: `D:\OpenCV\opencv\build\x64\vc16\lib`
- Dependency: `opencv_world4120d.lib`

默认入口在 `ObjectCounting/main.cpp`，默认读取：

- `D:/Projects/opencv-object-counting/assets/input/pill2.jpg`

程序运行后会弹出窗口显示结果，按键后退出。也可以在命令行传入图片路径覆盖默认值。

## 当前状态

如果你想看“更稳定的已完成版本”，应该回到较早的 `stable/HSV-counting` 思路；如果你想看“为什么想做鲁棒性、最后为什么没做成”，这个分支就是那次尝试本身。

它的价值主要在两点：

- 把处理流程从单文件试验代码整理成了可继续迭代的模块
- 留下了一个很明确的失败样本，说明传统规则链条在这类任务上为什么容易失效

## 后续如果要继续

比较合理的下一步不是继续硬调几个阈值，而是先补齐实验基础设施：

- 去掉硬编码路径，改成参数或配置
- 保存每一步中间结果，方便逐步定位失效点
- 为不同类别样本建立最小评测集
- 记录每张图的计数真值和误差
- 把阈值变成可配置项，而不是散落在代码里
- 再决定是否继续走传统视觉，还是转向学习方法
