# `archive/counting-v1` 实验笔记

## 这次分支想解决什么

`stable/hsv-counting` 那条线本质上还是单一路径方案：对一类样本有效，但一旦图片类型、颜色分布、粘连情况变化，方法就很容易失效。

这个分支的目标不是继续把一个 HSV 阈值调到更准，而是往“更像流程”的方向走一步：

- 先判断当前图片更适合看颜色还是看明暗
- 再做统一二值化
- 再判断是否存在粘连
- 只有在疑似粘连时才尝试分水岭
- 最后统一提轮廓、过滤、画框

换句话说，这个分支想做的是一版“带分支判断的传统视觉计数流程”。

## 当前代码落地到了哪里

这次尝试的主要实现已经从零散试验代码收到了 `Process.cpp` / `Process.h` 里，入口在 `main.cpp`。

当前主流程大致如下：

1. `judgeBasicType()`
   根据 HSV 的 S 通道均值，决定走 S 通道还是灰度图。
2. `basicImg2Binary()`
   用 Otsu 做第一次二值化。
3. `normalizeBinaryMask()`
   统一前景和背景极性，尽量保证白色是目标。
4. `judgeCrave()`
   用轮廓面积分布判断是否可能存在粘连。
5. `craveType()`
   对疑似粘连的图执行分水岭。
6. `contoursProcess()`
   提取轮廓，按面积过滤，并在原图上画框。

当前启发式阈值包括：

- `S_THRESHOLD = 53.2`
- `CRAVE_RATIO_THRESHOLD = 2.0`
- `minArea = rows * cols * 0.0005`

这些值现在都直接写在代码里，属于典型的“先跑通再说”的实验状态。

## 这次尝试真正推进了什么

虽然结果没有达到“鲁棒”，但这条线不是完全没价值，至少做成了几件事：

- 把原来单点试验函数整理成了相对清楚的处理流水线。
- 明确区分了三个问题：基础表征选择、粘连判断、目标计数。
- 把分水岭放进了一个完整流程里，而不是孤立地单独试。
- 轮廓过滤和结果绘制已经形成了可重复调用的模块。
- 工程目前可以在本机环境下成功编译。

也就是说，这个分支没有做成稳定方案，但把“问题结构”梳理出来了。

## 为什么最后还是失败了

失败不是因为某一行代码写错了，而是因为这条技术路线本身对样本分布太敏感。

### 1. 基础分支判断太粗

`judgeBasicType()` 目前只看平均 S 值。这个特征太粗糙了：

- 它不能可靠区分“颜色差异主导”还是“光照差异主导”
- 一张图里只要背景颜色、反光、高光比例变了，均值就会漂
- 对不同类别目标，S 均值的可解释性也不一致

结果就是：这一步看似自动，实际只是把人工经验换成了一个更隐蔽的阈值。

### 2. 二值化质量决定了后面几乎所有上限

Otsu 在简单样本上能工作，但对这类图像来说：

- 背景不均匀
- 反光明显
- 阴影存在
- 目标颜色和背景有时并不稳定分离

只要第一张 mask 不稳定，后面的轮廓判断和分水岭就都会被拖垮。

### 3. 粘连判断启发式不稳

`judgeCrave()` 现在基于“最大轮廓面积是否显著大于平均面积”来判断粘连，这个规则非常脆弱：

- 目标数量一变，平均面积就会变
- 噪声一多，平均面积又会变
- 本来就存在尺寸差异的目标会误触发
- 单个大目标也可能被错判成粘连

它对一些样例可能有效，但很难成为真正可靠的判断条件。

### 4. 分水岭不是补救万能药

把分水岭接入流程以后，最大的体感是：

- 前面分得好，它锦上添花
- 前面分得差，它放大错误

传统分水岭非常依赖前景、背景和 unknown 区域的质量。当前这套 markers 生成方式还不足以让它在复杂样本上稳定工作。

### 5. 没有评测体系，就很难知道“到底稳了没有”

这个分支最大的工程短板其实不是算法，而是实验方式：

- 没有固定评测集
- 没有每张图的真值计数
- 没有批量跑图
- 没有中间结果留档
- 没有误差统计

所以很多时候只能凭“这张图看起来不错”来判断，而这很容易高估方法的泛化能力。

## 这次分支最后得到的结论

当前最真实的结论不是“参数还差一点”，而是：

**这类传统规则链在这个任务上高度依赖样本分布，一类图片往往就像一类新题。**

更直接一点说：

- 规则可以做出能跑的 demo
- 但规则很难自然长成真正的鲁棒性
- 多加几层判断，不等于问题就被普适地解决了

这个分支失败的地方，恰好说明了为什么“继续堆启发式”通常收益越来越低。

## 当前工程层面的遗留问题

除了方法本身，工程上还有几件事会持续阻碍实验：

- 图片路径写死在 `main.cpp`
- OpenCV 本地路径写死在 `.vcxproj`
- 没有统一配置文件
- 没有保存中间图像的调试机制
- 没有批处理入口
- 没有自动化测试或最小评测脚本

这些问题不一定导致算法失败，但会让“定位失败原因”变得很慢。

## 如果以后重开这个方向

如果以后还想继续这条线，比较合理的顺序应该是：

1. 先把路径、阈值、样本列表全部参数化。
2. 给每一步保存中间结果，而不是只看最终框图。
3. 建一个最小评测集，至少覆盖 pills 和 coins 两类样本。
4. 给每张图记录目标真值，开始真正算误差。
5. 再决定是继续做传统视觉分治，还是转学习方法。

如果继续坚持传统方法，也更建议：

- 先按样本类别分别建立 baseline
- 再考虑能不能抽象出共用逻辑
- 不要一开始就假设“一套规则通吃所有图”

## 总结

这次分支的价值，不在于它做成了鲁棒计数，而在于它把失败原因暴露得很清楚：

不是代码还差一点，而是问题本身已经开始逼近传统启发式方法的边界了。
