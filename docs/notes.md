# `exp/codex-test-fail` notes

## 这条分支到底在试什么

这条分支不是普通的功能开发，而是一次很明确的实验：

- 让 Codex 主导代码生成和结构推进
- 尽量不借助学习方法
- 看传统图像处理到底能被推到多远

我想验证的不是“能不能把 demo 跑起来”，而是：

**纯 AI coding 能不能把一套传统视觉方案堆到足够稳，至少能覆盖多类型图像。**

现在回头看，这个分支给出的答案是：

**能更快把系统堆复杂，但不能自动把方法变对。**

## 这条分支的实际演进

如果只看最终代码，很容易误以为这里只做了一次文档化重构。实际不是，这条分支经历了两层推进。

### 第一层：把早期规则链模块化

这一层主要体现在：

- `Process.cpp`
- `Process.h`
- `main.cpp`
- `TrackbarTest_main.cpp`

这一版的主线是：

1. 用 `judgeBasicType()` 判断走 S 通道还是灰度图
2. 用 `basicImg2Binary()` 做第一次二值化
3. 用 `judgeCrave()` 判断是否疑似粘连
4. 用 `craveType()` 尝试拆粘连
5. 用 `contoursProcess()` 过滤轮廓并画框

这一层其实已经能说明很多事了：

- Codex 很擅长把零散试验拼成有函数边界的流程
- 它也很擅长把“这里再加个判断”迅速落成代码
- 但它依然没有让传统规则天然获得泛化能力

### 第二层：Codex 重新写了一版更重的 pipeline

真正属于 `exp/codex-test-fail` 这个名字的，是最后这套：

- `CodexProcess.cpp`
- `CodexProcess.h`
- `CodexPipeline_main.cpp`
- `assets/output_codex/`

这不是简单修修补补，而是另起了一条更激进的传统视觉线。

当前默认 x64 构建实际跑的是这条 Codex pipeline，不是旧的 `main.cpp`。

## Codex 版 pipeline 实际做了什么

这套流程比早期 HSV / Otsu 路线复杂得多，核心步骤大概是：

1. 从图像边缘采样，估计背景的 Lab 颜色。
2. 计算每个像素和背景之间的色度距离。
3. 再叠加局部亮度残差和饱和度，形成一张前景分数图。
4. 对分数图做 Otsu，得到初始 mask。
5. 用形态学、填洞和连通域面积过滤清理 mask。
6. 对每个连通域单独估计“它可能包含几个目标”。
7. 用腐蚀种子或距离峰值种子，在局部 mask 内做 seeded growth。
8. 最后再按面积、填充率和轮廓提取检测结果。

这版还有一个很重要的区别：

它不只是给最终结果，而是把中间过程几乎都存下来了。

对每张输入图，都会保存：

- `score`
- `raw_mask`
- `clean_mask`
- `split_mask`
- `markers`
- `result`

现在 `assets/output_codex/` 里有 54 张图，正好对应 9 张输入图各自的 6 类中间结果。

从实验角度看，这一步其实很有价值，因为它让失败不再只是“看起来不准”，而是能看到到底是哪一步开始坏掉。

## Codex 在这条分支上到底做成了什么

如果只从“推进速度”和“工程组织”看，这次实验其实挺成功：

- 它把一堆松散想法很快组织成了能编译的系统
- 它补出了批处理入口，不再只盯一张图
- 它补出了中间结果保存，调试信息比早期版本完整很多
- 它把旧版规则链和新版 Codex pipeline 并排留在一个工程里，方便比较

也就是说，**Codex 对“把实验做完整”这件事帮助很大。**

它让这条线更快从“拍脑袋想法”进入“可以被验证、可以被否定”的状态。

## 但它为什么还是失败了

失败的关键不是“代码没有写出来”，而是这条方法路线本身还是建立在大量前提之上。

### 1. 背景假设依然太强

Codex 版 pipeline 的第一步就是从边缘估计背景颜色。

这一步在背景比较统一时很有用，但它其实默认了：

- 图像边缘大多是背景
- 边缘不被目标占据
- 边缘没有太强阴影或杂物

只要这些前提被打破，后面的前景分数图就会从起点开始偏掉。

### 2. 分数图再复杂，本质也还是手工规则

这版确实比早期只看 HSV 强很多，因为它把：

- Lab 色度距离
- 局部亮度残差
- 饱和度

叠成了一张前景分数图。

但说到底，这仍然是人工设计特征和权重。

也就是说：

- 它比单一阈值聪明
- 但它还没有“学会”什么是目标
- 它只是在更复杂地描述“我猜目标应该长这样”

当目标类别变多时，这种猜测会越来越不稳。

### 3. 后面的拆分依赖前面的 mask 正确

Codex 版里最有技术味的一段，是局部 seed 估计和 seeded growth。

这套逻辑看起来已经很像一个完整的 instance split 流程了，但它依然有一个很硬的限制：

**它只能在已有前景区域大致正确的前提下继续细分。**

如果前面的 raw mask / clean mask 已经错了，那么：

- 种子会种错
- growth 会长错
- 最终检测自然也会错

所以后面的复杂拆分，并没有从根本上拯救前面的错误。

### 4. 多类型图像会让启发式很快失去统一性

这个分支里已经能看到一个很清楚的事实：

- 药片和硬币就已经不是同一类统计分布
- 颜色、材质、边缘、反光和背景都不一样
- 一套在 pills 上看起来合理的规则，未必还能解释 coins

更不要说如果继续引入更多目标类别，问题只会变得更糟。

所以这里真正失败的，不是某个阈值，而是“希望一套共享规则覆盖多类型图像”这个前提。

## 这次实验对 AI coding 的真实结论

这条分支让我比较确信一件事：

**AI coding 很擅长加速实现，但不会自动越过方法边界。**

它可以很快帮我做到：

- 把想法写成代码
- 把代码拆成模块
- 把模块串成 pipeline
- 把 pipeline 跑成一批输出图

但如果底层方法本身不够表达这个任务，那么：

- 代码再多
- 逻辑再复杂
- 中间图再全

最后也只是更快地证明“这条路走不通”。

这不是 Codex 的失败，反而是它的价值之一：

**它帮我更快做实了这次失败。**

## 这条分支最终留下的结论

如果把这次实验的结论压缩成一句话，就是：

**纯 Codex / AI coding 能把传统视觉方案推到更复杂、更完整，但依然不足以让单套图像处理规则稳定识别多类型图像。**

再直接一点：

- AI 可以加速编码
- AI 不会自动修复错误的方法假设
- 传统视觉可以做出局部有效的 pipeline
- 但很难自然长成跨类型稳健识别

## 如果以后还继续这个项目

如果以后还继续，我觉得顺序应该换一下：

1. 先按类别建最小评测集。
2. 先记录真值和误差，而不是只看图片观感。
3. 先分清哪些类别还能继续做传统视觉。
4. 对真正跨类型的目标，尽早转向学习方法。
5. AI coding 继续用，但把它当成实验加速器，不要把它误当成方法突破器。

## 一句总结

`exp/codex-test-fail` 不是一个成功版本，但它是一条很有价值的失败分支。

它说明了两件事：

- Codex 确实能把实验推进得很快。
- 但光靠传统图像处理，确实很难把多类型图像识别做稳。
